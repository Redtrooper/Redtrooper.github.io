Project Description

Astreya is an endless top-down shooter created in C++ using where the player fights an onslaught of different enemies while trying to get the highest score possible.

About

Astreya is a game styled after arcade games such as Asteroids & Galaga. By basing the gameplay loop on the arcade style, the project scope has remained small allowing for a more complete experience. The game was made with replayability in mind through the high score system as well as player upgrades which both enable different playstyles. This project was made while at Full Sail University for my Project & Portfolio V class. Additionally, this project was made using Gateware (a graphics, sound, and input library), Vulkan (a graphics API), and EnTT (an entity component system).

The Gameplay Loop

When making the gameplay loop, I wanted to focus on replayability and ramping difficulty to make each run unique and fun. To do so, I made sure that the game only ends when the player loses all of their health and that more enemies spawn over time. I also wrote the portion of the game's code that deals with enemy spawn position, allowing for each enemy to spawn randomly around the screen border. Additionally, I balanced the enemy spawns by increasing the number of asteroids that can spawn over time in addition to the UFO & Noremac which spawn based on time and the Vampire which spawns based on score. Last, I ensured that the player's base stats were balanced with the enemies so the game would have a reasonable difficulty curve.

Upgrades

The largest difference between Astreya and the games that it was inspired by is the upgrades system. Upgrading stats serves as the main way for the player to get stronger alongside the enemies while also allowing "builds" that tailor the gameplay to the style of each player. The most important part of the system is that each upgrade cost score, creating a drawback to the system. While the pricing goes up exponentially, the actual impact of most upgrades (excluding those that have static increases ex. health or guns) uses a diminishing returns algorithm to keep the gameplay balanced. With the player being an entity component, the system grabs that entity to specifically increase the stats that are selected. The player uses the stats that are created to add multipliers to in game actions (like speed or fire rate), the user interface calls the function necessary to increase those multipliers, and they are reset once the player dies.

High Scores

To keep the game competitive, I implemented an online high score API provided by an instructor at Full Sail University. This API originally stored ints as strings online, then converted them back to ints for saving scores. I modified that code to save both the player's score and their name (as the online component simply stored a list of strings). The version of the code that this project uses saves the name and the score with a space used as a delimiter. Then with an updated version of the parsing function we parse the name and score into the game to display to all players. With high scores being the competitive part of our game, they are saved fully online without any local data used to keep players from interfering with the leaderboards. This ensures the integrity of the leaderboards while keeping the game competitive.

Next Steps

Going forward, I plan to refactor some of the existing code in the game for the purposes of future iteration. Specifically, refactoring some of the UI code to allow for more streamlined menu creation and navigation. Next, to add more challenge to the later portion of the game I will increase the health of the enemies the longer the player survives. Finally, I will integrate the Steamworks SDK to allow for a full release on Steam. Overall, I want to polish what we have complete to prepare for release as the game already feels very complete due to its small scope.
