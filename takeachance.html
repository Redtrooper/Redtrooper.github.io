Take A Chance

  • Role: Co-Lead Developer
  • Studio: Quantex Games
  • Team Size: 2
  • Duration: 2 Months
  • Unity: https://unity.com

Project Description

Take A Chance is a rougelike level-based first-person shooter developed using Unity.

About

Take A Chance was the first game that I ever created and was made as my midterm project at Full Sail University. While it reamins mostly unfinished, in creating as much as I did I learned a lot about the game creation pipeline. The idea that my team had going in was a replayable first-person-shooter that was inspired by elements from games such as Risk of Rain 2. Overall, the game stands as a solid foundation for starting with and ideas from that game helped me move forward to becoming a gameplay programmer.

Weapon System

When making Take A Chance I created a combat sytem that allows the player to switch between 2 different weapons. Both weapons are ranged, one being an assault rifle and the other a shotgun with bullet spread. Creating this system allowed me to learn about projectile-based combat (the creation of a projectile which handles the damage to what it hits) and how to create a basic weapon inventory. I also solved the issue of weapons not firing towards the center of the screen by rotating the weapon towards center screen to ensure accuracy. Overall, this weapon system established many core components that would go into the creation of the weapon system for my next major shooter Mechanical Meltdown.

Items

The core component of Take A Chance was the creation of the item system. Each item spawns randomly, allowing players to have a different experience each time. While some items directly alter stats that the player has, others (such as grenade and air dash) unlock abilities that the player can use during combat. I created each item to enhance player stats, however in not using an algorithm (such as a diminishing returns algorithm) to balance the item stat increases many items can become incredibly overpowered or game breaking after collecting too many. This is part of the reason why in my next game Astreya I used a diminishing returns algorithm to keep the game balanced.

Game States

The last core component of the game I worked on was game states. In creating a level-based game, my team ran into the problem of saving player data between levels. Our game loop required the player to be able to bring their weapons and items between levels to continue progression. To enable the player to do this, I used the Unity PlayerPrefs system. This system allowed me to cache off the player's progress and items to be reloaded in the next level. However, this causes another issue in the player being able to alter those files to change their game progress. In a singleplayer game this isn't much of an issue, but a multiplayer game or a more competitive game wouldn't use a system this way to ensure the game remains fair. This is another line of reasoning that was carried into my next project, Astreya, as I ensured that there was no way to edit online scores before they were sent to the server.

Postmortem

Completing this project taught me a lot about game creation that I carried forward into my next projects. In being one of the most tough games to create because the team went down to 2 people, and being the first game I made it still is special to me. Although I've suspended any development on the project I wouldn't be fully against returning to it in the future with the knowledge I've gained since then. This project taught me a lot about prioritizing what is important, communicating with team members, and about resolving conflicts that arise during development. I think that the game came out incredibly well for what it was, although there were many features that had to be cut. The biggest lesson I learned is to be careful of scope and time, because with our team size in mind I should've cut features such as procedural generation earlier to allow more time to other areas of the game.
